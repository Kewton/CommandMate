## 詳細所見

### Must Fix (1件)

#### SEC-001: sync APIの500エラーレスポンスで内部エラーメッセージがクライアントに露出する

**重要度**: must_fix
**OWASP**: A01:2021 - Broken Access Control / Information Disclosure

POST /api/repositories/sync の catch ブロック（route.ts 63-70行目）で、`error.message` をそのままクライアントに返している。ファイルシステムパス、DB接続エラー、権限エラー等の内部情報がエラーメッセージに含まれる可能性がある。

```typescript
// 現行の実装（問題あり）
const errorMessage = error instanceof Error ? error.message : 'Failed to sync repositories';
return NextResponse.json({ error: errorMessage }, { status: 500 });
```

設計書セクション7のセキュリティ設計ではこの問題に言及していない。

**対応推奨**: 設計書のセキュリティ設計セクションに、エラーレスポンスにおける情報漏洩防止方針を追記すること。実装時はクライアントへのレスポンスには汎用メッセージのみを含め、詳細はサーバーログに出力する。なお、これは既存APIの問題であり、本IssueのSyncButton実装自体は既存APIをそのまま呼び出すだけであるため、SyncButton側での対応は不要。設計書での認識表明が主な対応となる。

### Should Fix (3件)

#### SEC-002: CSRF保護メカニズムの具体的な説明が不足

**重要度**: should_fix
**OWASP**: A01:2021 - Broken Access Control

設計書セクション7で「Next.js API Routesの既存CSRF保護を継承」と記載しているが、Next.js API Routes自体にビルトインのCSRFトークン検証機構は存在しない。

実際の保護メカニズムは以下の3層で構成される:
1. **SameSite=strict Cookie**: auth.tsの`buildAuthCookieOptions()`で設定。クロスサイトリクエストではCookieが送信されない
2. **Content-Type検証**: api-client.tsの`fetchApi()`がContent-Type: application/jsonを設定。HTMLフォームからのPOSTではapplication/jsonを送信できないため、CSRF攻撃が困難
3. **同一オリジンポリシー**: ブラウザのCORS制御により、異なるオリジンからのfetchリクエストは制限される

**対応推奨**: セキュリティ設計のCSRF項目を上記3点を根拠とした正確な記述に修正すること。

#### SEC-003: SyncButton連打・自動化によるDoS的なリクエストへの考慮不足

**重要度**: should_fix
**OWASP**: A04:2021 - Insecure Design

`isSyncingRef`によるクライアントサイドの二重実行防止は記載されているが、ブラウザ開発者ツールや自動化スクリプトでの直接API呼び出しに対するサーバーサイド保護が設計書で言及されていない。sync APIはファイルシステムスキャンを伴う比較的重い処理である。

**対応推奨**: セキュリティ設計セクションに、サーバーサイドレート制限の必要性について方針を記載すること。現時点でローカル開発ツールとしての利用を前提とし実装不要と判断する場合も、リスク認識と将来Issue化の方針を明記する。

#### SEC-004: sync APIレスポンスにファイルシステムパスが含まれる

**重要度**: should_fix
**OWASP**: A01:2021 - Broken Access Control / Information Disclosure

SyncResponseのrepositories配列にサーバーのファイルシステム絶対パスが含まれている。SyncButtonではworktreeCountのみを使用しているが、レスポンス全体がクライアントに到達する。

**対応推奨**: 設計書のセキュリティ設計セクションに、SyncButtonがworktreeCountのみを使用していること、repositories配列のパス情報はクライアント側で参照しないことを明記する。パス情報除去は既存API側の改善として別Issue化を検討。

### Consider (2件)

#### SEC-005: i18nテンプレート変数の安全性記述の明確化

Toastメッセージの`{count: result.worktreeCount}`はサーバー由来の数値であり、ReactのJSXエスケープとnext-intlのICU MessageFormatによる二重保護があるため、XSSリスクは実質ゼロ。設計書の記述は正しいが、保護メカニズムをより具体的に記載すると明確になる。

#### SEC-006: 認証無効環境での動作の明記

CM_AUTH_TOKEN_HASH未設定時（認証無効環境）でもSyncButtonが正常動作することを設計書に補足的に記載すると、実装者の理解が深まる。
