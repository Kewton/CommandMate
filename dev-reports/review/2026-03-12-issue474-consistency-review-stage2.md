## Must Fix (3件)

### S2-M1: resolveAndValidateRealPath 関数のシグネチャ不一致（引数数・用途）

**設計書の記述** (セクション6):
```typescript
const absoluteImagePath = resolveAndValidateRealPath(
  worktreePath,
  body.imagePath,
  '.commandmate/attachments'  // 許可サブディレクトリ
);
```

**実際のシグネチャ** (`src/lib/path-validator.ts` 161行目):
```typescript
export function resolveAndValidateRealPath(targetPath: string, rootDir: string): boolean
```

- 実際の関数は **2引数** で **boolean** を返す
- 設計書は **3引数** で **string | null** を返す前提で記述
- 第3引数の「許可サブディレクトリ」制限機能は実装に存在しない
- 引数の意味も逆転: 実際は第1引数が検証対象、第2引数がルート

**修正提案**: 実際のシグネチャに合わせて設計書を修正する。サブディレクトリ制限が必要な場合は、`resolveAndValidateRealPath` 呼び出し後にパスプレフィックスの追加チェックを行うロジックを設計に明記する。

### S2-M2: isPathSafe 関数の引数順序が設計書で逆転

**設計書の記述** (セクション6):
```typescript
if (!absoluteImagePath || !isPathSafe(worktreePath, absoluteImagePath)) {
```

**実際のシグネチャ** (`src/lib/path-validator.ts` 46行目):
```typescript
export function isPathSafe(targetPath: string, rootDir: string): boolean
```

**既存の正しい呼び出し** (`upload/route.ts` 117行目):
```typescript
if (!isPathSafe(normalizedDir, worktree.path)) {
```

設計書では `isPathSafe(worktreePath, absoluteImagePath)` と呼び出しており、ルートディレクトリを検証対象として渡している。これではパストラバーサル防御が正しく機能しない。

**修正提案**: `isPathSafe(absoluteImagePath, worktreePath)` に修正する。

### S2-M3: fetchApi ラッパーの Content-Type 強制による uploadImageFile 実装制約

`api-client.ts` の `fetchApi` 関数（47行目）は全てのリクエストに `Content-Type: application/json` を強制設定する。設計書セクション4で追加予定の `uploadImageFile` メソッドは `FormData` を送信する必要があり、`Content-Type` をブラウザに自動設定させる（`multipart/form-data` + boundary）必要がある。`fetchApi` をそのまま使用できない。

**修正提案**: `uploadImageFile` は `fetchApi` を使わず直接 `fetch` を呼び出す実装とする旨を設計書に明記する。

### S2-S2: API 名前空間の不一致: apiClient vs worktreeApi

設計書セクション7で `apiClient.uploadImageFile` と記述しているが、既存コードでは `worktreeApi` 名前空間が使用されている（`MessageInput.tsx` でも `worktreeApi.sendMessage` を呼び出し）。

**修正提案**: `apiClient` を `worktreeApi` に修正する。

---

### S2-S3: upload API のバリデーション方式の不整合

設計書では `image-extensions.ts` の定数を画像バリデーションに使用する方針だが、既存の `upload/route.ts` は `uploadable-extensions.ts` の関数群（`isUploadableExtension`, `validateMagicBytes`, `validateMimeType`）を使用している。`uploadable-extensions.ts` には既に SVG 除外済みの画像拡張子が含まれており、`ATTACHABLE_IMAGE_EXTENSIONS` を別途作成する必要性が薄い可能性がある。

**修正提案**: upload API のサーバーサイドバリデーションは既存の `uploadable-extensions.ts` をそのまま活用し、`ATTACHABLE_IMAGE_EXTENSIONS` は UI 側の accept 属性とクライアント側フィルタリング専用と位置づけることを設計書に明記する。

---

### S2-S4: send/route.ts への imagePath 挿入位置の未詳細化

既存の `send/route.ts` は複雑な処理フロー（worktree検証 -> body解析 -> content検証 -> CLIツール選定 -> セッション起動 -> pending応答保存 -> orphan検出 -> メッセージ送信 -> DB保存 -> ポーリング開始）を持つ。`imagePath` バリデーションをどの位置に挿入するかが設計書のコード例からは不明確。

**修正提案**: `imagePath` のパスバリデーションは CLIツールのセッション起動前（早期リターン）に行うべき旨を明記する。

---

### S2-S5: sendMessage シグネチャ変更の影響範囲が未記載

設計書はシグネチャを `sendMessage(id, content, options?)` に変更する方針だが、影響を受ける呼び出し箇所のリストが記載されていない。現在は `MessageInput.tsx` の1箇所のみだが、明示しておくべき。

**修正提案**: セクション4またはセクション9に影響箇所リストを追記する。
