CommandMate User Scenarios / Journey Map

既存機能を前提に、ユーザシナリオをライフサイクルで再整理したページです。 「どこで登録し、どこで選び、どこで実行し、どこでレビューし、どこで管理するか」を Desktop / Mobile で同じ mental model に揃えています。

原則 1 Home は状況判断と Inbox
原則 2 Repositories は登録・同期・削除・復元
原則 3 Sessions は worktree 選択と探索
原則 4 実行は選択済み worktree detail で行う
原則 5 More は設定と外部連携

North Star

現状はホームに Repository 管理、Worktree 一覧、External Apps が並び、シナリオごとの入口が混在しています。 新UXでは、ユーザーの行動を 導入 → 登録 → 選択 → 実行 → レビュー → 管理 の流れに沿って分離します。

Start / Login → Repositories → Sessions → Worktree Detail → Review / Home
整合性の核
  • リポジトリ登録は Home 直置きではなく Repositories に集約
  • 新規タスク送信はグローバルではなく 選択済み worktree で行う
  • Waiting / Completed / Error は Home に集約
  • 検索・フィルタ・選択は Sessions に寄せる

1. ユーザシナリオ棚卸し

フェーズ ユーザシナリオ 正しい入口 主な完了条件

登録 ローカルディレクトリ指定で repository 登録 `Repositories > Add > Local Path` worktree が認識される
登録 Clone URL 指定で repository 登録 `Repositories > Add > Clone URL` clone 完了後に自動登録
登録 Sync で repository / worktree を再検出 `Repositories` 一覧が最新になる
探索 検索 / filter / favorite / status で worktree を探す `Sessions` 対象 worktree が選択できる
実行 worktree を開きメッセージ送信 `Sessions > Worktree Detail` agent が走り始める

実行 Auto Yes を有効化 `Worktree Detail` 無人実行が始まる
レビュー 実行中タスクの監視 `Home` / `Sessions` / `Worktree Detail` 状況把握できる

3. シナリオ別の導線整理

Setup

初回導入 / ログイン

`commandmate start` 後、必要なら token login を通過。認証後の初回表示は Home(empty state) に統一し、ここから `Add Repository` を強く出す。

Setup

ローカルディレクトリ登録

`Repositories` で `Add Repository` を開き、`Local Path` を選択。 Scan 結果と登録完了は同一フロー内で見せ、終了後 `Sessions` へ誘導する。

Setup

Clone URL 登録

`Repositories` の `Clone URL` に統一。clone 中は別画面に飛ばさず、 progress / pending / failed を同コンテキストで表示する。

Run

worktree 選択

選択、検索、favorite、status 絞り込みはすべて `Sessions` に集約。 Home に一覧を再実装しない。

Run

メッセージ送信 / 実行開始

実行開始は必ず 選択済み worktree detail で行う。 グローバルな “Send” 画面は作らず、対象 worktree が曖昧にならないようにする。

Run

Prompt への応答

最短導線は `Home(Inbox)`。詳細確認が必要なら `Worktree Detail` へ遷移。 “すぐ返す” と “文脈を見て返す” の両方を許容する。

Review

実行中の監視

概況は `Home`、詳細は `Sessions / Worktree Detail`。 Home はリアルタイムの全体把握、Sessions は探索、Detail は深掘りに役割分担する。

Review

完了後の確認

`Home` に Completed queue を置き、そこから diff / logs / notes / files へ接続。 完了したが未確認のものを埋もれさせない。

Manage

Repository 削除 / 復元 / 外部連携

Repository 関連は `Repositories`、外部アプリは `More` に集約。 管理操作を Home の主文脈から切り離す。

4. 整合性ルール

Rule 1 — One job, one place

Repository 登録を Home / Modal / 空状態で分散させず、基本は `Repositories` に一本化。 例外は初回 empty state からのショートカットのみ。

Rule 2 — Home is for decisions

Home は “何をするか決める画面”。Waiting、Completed、Error、Running summary を置く。 管理フォームは常設しない。

Rule 3 — Sessions is for finding

一覧、検索、絞り込み、favorite、status 操作は Sessions に集約。 Home のカードは検索用ではなく、優先タスク用に絞る。

Rule 4 — Detail is for acting

実際の送信、Auto Yes、ファイル閲覧、ノート、ログ閲覧は worktree detail に集約。 “何に送るか分からない send” を作らない。

Rule 5 — Review is a first-class state

完了タスクを “終わったら見えなくなる” 状態にしない。Home / Review から再訪しやすくし、 次の判断につなげる。

Rule 6 — Mobile mirrors desktop

画面数は減らしても、役割は揃える。Home は Inbox、Sessions は探索、Repos は登録、More は管理。

5. 実装に落とした時の差分

現状 変更後 狙い
Home に RepositoryManager / WorktreeList / ExternalApps が縦積み Home は Mission Control、Repository 管理と Apps を分離 初見の価値訴求を明確化
Desktop は sidebar、Mobile は homepage list で worktree 選択 両方とも `Sessions` に探索機能を寄せる device 間で役割を一致
Send の文脈が detail 依存で、Home からは見えにくい 「送信は detail」と明示し、Home には対象 worktree への導線だけ置く 誤送信防止と mental model 明確化
Excluded Repositories が一覧末尾に出る `Repositories > Excluded` に昇格 削除と復元を同じ文脈に収める
External Apps が Home 下部 `More > External Apps` へ移動 主導線を汚さない
