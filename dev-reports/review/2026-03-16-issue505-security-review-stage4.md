致命的な脆弱性はないが、3件の should_fix と2件の nice_to_have を推奨する。

## 3. 指摘事項

### DR4-001 [should_fix] rehype-sanitize カスタムスキーマが data:/vbscript: スキームを許可する

**カテゴリ**: A03:2021-Injection / XSS

**問題**: 設計方針書 Section 4-2 のカスタムスキーマは href 属性に `/^(?!javascript:).*$/` を使用し、`javascript:` のみを拒否するブロックリスト方式である。`data:text/html,<script>alert(1)</script>` や `vbscript:` など他の危険スキームが通過する。

**推奨**: 許可リスト方式に変更する。`http:`, `https:`, `mailto:`, `tel:`, `#`、および相対パスのみを許可する。あるいは rehype-sanitize のデフォルトスキーマ（既に安全なプロトコルのみ許可）をカスタマイズせず使用する方がより安全。

**影響ファイル**: `src/components/worktree/MarkdownPreview.tsx`, `src/lib/link-utils.ts`

---

### DR4-002 [should_fix] HtmlSourceViewer の dangerouslySetInnerHTML が highlight.js 出力に依存

**カテゴリ**: A03:2021-Injection / XSS

**問題**: 既存の `HtmlSourceViewer`（HtmlPreview.tsx line 74）は `dangerouslySetInnerHTML={{ __html: lines[idx] }}` を使用している。Issue #505 自体はこのコードに触れないが、postMessage 経由で受け取った href がソースビューにレンダリングされる経路がないことを実装時に確認する必要がある。

**推奨**: postMessage の href がソースビューに反映されないことを実装時に確認すること。

**影響ファイル**: `src/components/worktree/HtmlPreview.tsx`

---

### DR4-003 [should_fix] handleLinkFromIframe の入力バリデーション不足

**カテゴリ**: A01:2021-Broken Access Control / Open Redirect

**問題**: `classifyLink()` のロジック自体は安全（`javascript:` は `relative` 扱いとなり `resolveRelativePath` で null 返却）だが、postMessage 経由の href に対する入力サニタイズ（最大長チェック、制御文字排除）がない。

**推奨**: `handleLinkFromIframe` で `classifyLink` 呼び出し前に以下を追加:
1. 最大長チェック（2048文字）
2. 制御文字の排除
3. null/undefined ガード

**影響ファイル**: `src/lib/link-utils.ts`, `src/components/worktree/HtmlPreview.tsx`

---

### DR4-004 [nice_to_have] postMessage リスナー登録タイミングの最適化

**カテゴリ**: A05:2021-Security Misconfiguration

**問題**: 現状の多層防御で十分だが、postMessage リスナーの登録タイミングを iframe ロード完了後に限定することで、iframe ロード前の競合メッセージを排除できる。

**推奨**: useEffect の依存配列に iframe ref を追加することを検討。

---

### DR4-005 [nice_to_have] onOpenFile 呼び出しへの debounce 追加

**カテゴリ**: A07:2021-Identification and Authentication Failures / DoS

**問題**: 大量のリンクを含む Markdown での連続クリックが API 連続リクエストを発生させる可能性。MAX_FILE_TABS=30 でタブ数は制限されるが、API レート制限はない。ローカルツールのためリスクは低い。

**推奨**: onOpenFile に 300ms 程度の debounce を追加（UX改善にもなる）。

## 5. iframe セキュリティ分析

| モード | sandbox 属性 | リンク対応 | リスク |
|--------|-------------|-----------|--------|
| Safe | `sandbox=""` | なし | NONE |
| Interactive | `sandbox="allow-scripts"` | postMessage経由 | LOW |

**Sandbox Escape リスク**: LOW -- `allow-same-origin` を付与しないため iframe 内スクリプトが sandbox 属性を変更することは不可能。

**Clickjacking リスク**: NONE -- iframe は srcDoc で表示、外部サイト埋め込みなし。ローカルツールのため対象外。
